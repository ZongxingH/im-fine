# Skill: harness-audit

## Trigger

Use when evaluating imfine demo quality, true-harness credibility, or the observability of a run.

This skill adapts AHE's Harness Audit, evidence-driven change manifest, and falsifiable verification principles for imfine.

## Inputs

- Current run id or active `.imfine/state/current.json`.
- `.imfine/runs/<run-id>/` artifacts.
- Runtime story/debug status if available.
- Demo summary report if available.

## Steps

1. Identify the run under review.
2. Build an artifact inventory:
   - Agent-authored: `agents/*/handoff.json`, `agents/*/output.md`, `agents/*/commands.md`, acceptance matrix authored by agents.
   - Runtime-derived: dispatch contracts, provider receipts, wave history, quality lineage, runtime requirements, final gates, true harness evidence, trace files, archive report.
3. Check true-harness closure:
   - Orchestrator declaration is valid.
   - Required dispatch contracts exist.
   - Required provider-origin receipts exist and are completed.
   - Handoffs exist and reference real evidence.
   - Parallel execution has completed waves for required contracts.
   - Runtime final gates exist and were generated by `imfine-runtime`.
   - `true-harness-evidence.json` is fresh and `true_harness_passed=true`.
4. Check demo observability:
   - `status --story` or equivalent summary separates Agent-authored and Runtime-derived evidence.
   - Primary blocker is visible when the run is blocked.
   - Debugger claims cite artifacts or trace refs.
   - Gate phase is concise enough for a demo but not hiding failed gates.
5. Classify demo quality as `pass`, `pass_with_risks`, `blocked`, or `misleading_demo`.
6. For each issue, record:
   - failure evidence
   - root cause
   - targeted fix
   - predicted impact
   - regression risk

## Outputs

- Demo observation report, preferably `.imfine/runs/<run-id>/analysis/demo-observation.md`.
- Short user-facing summary with verdict, primary evidence, and next owner.

## Failure Handling

If required artifacts are missing, mark the demo `blocked` or `misleading_demo` instead of inferring success.

If the current provider cannot launch an independent Harness Auditor Agent, the current session may run this skill directly, but the report must disclose `auditor_execution=single_session_skill`.

## Prohibited

- Do not mutate delivery artifacts while auditing.
- Do not accept runtime-only receipts as native subagent proof.
- Do not approve demo substitutes unless QA or Reviewer evidence explicitly accepts them.
- Do not rely on screenshots or provider UI observations as completion proof.
